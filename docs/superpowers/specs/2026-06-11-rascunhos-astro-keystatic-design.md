# rascunhos.blog → Astro + Keystatic (light CMS) — Design Spec

**Date:** 2026-06-11
**Status:** Draft for review
**Repo:** `hhpmacedo/rascunhos_blog` · branch `feat/astro-keystatic-cms`

## Goal

Give rascunhos.blog a self-service way to **add *and edit*** posts from any device,
without changing how the site looks or breaking any existing URL. Today the site is
hand-built static HTML generated by `publish.js`, which (a) is add-only and (b) discards
the markdown source, so there is nothing to edit. This migration introduces a real
markdown source of truth and a hosted editor.

## Approach (decided)

Port the site to **Astro** (`output: 'static'` + Vercel adapter) reproducing the current
**v2 design**, with posts stored as `src/content/posts/*.md` (markdown + frontmatter), and
add **Keystatic in GitHub mode** at `/keystatic` as the hosted admin. Public pages stay
static; only Keystatic's routes run on-demand. This mirrors the validated pattern from the
`hugo-website` Keystatic build, including its rendered-HTML equivalence verification gate.

Decisions already made:
- **Access model:** hosted admin, any device, commits via GitHub.
- **Post URLs:** preserve **exact** `/posts/<slug>.html` paths (zero breakage for inbound
  links, bookmarks, RSS guids, SEO).
- **Sequencing:** the v2 redesign is shipped first (done, `origin/main` @ `913b4a1`); this
  port reproduces that final design.

## Architecture

```
src/
  content/
    posts/<slug>.md          # source of truth: frontmatter + markdown body
    config.ts                # content collection schema (Zod)
  layouts/Base.astro         # <head>, sidebar, footer, analytics, subscribe float
  pages/
    index.astro              # homepage: featured + recent + archive rhythm + pagination pg 1
    page/[page].astro        # pagination pages 2..N
    posts/[slug].html.astro  # post page — note the literal ".html" in the route
    o-culpado/index.astro    # author page
    rss.xml.ts               # RSS feed (@astrojs/rss)
  styles/styles.css          # the EXISTING css, reused as-is
keystatic.config.ts          # Keystatic collection + GitHub storage
astro.config.mjs             # static output + Vercel adapter + react() + keystatic()
```

The current hand-built `site/` tree and `publish.js` are **removed** once the Astro build
produces byte-equivalent output (see Migration + Verification). `styles.css` moves verbatim.

## Content model

`src/content/config.ts` collection `posts`, schema:
- `title: string` (required) — rendered as the `<h1>` (kept lowercase as today).
- `pubDate: date` (required) — drives ordering and the `data-date` attribute.
- `excerpt: string` (optional) — homepage/RSS summary. If omitted, auto-derived from the
  body (first ~280 chars, same rule as today's `publish.js`), so editing stays low-effort.
- `draft: boolean` (default false) — committed but hidden from the live site/feed.

Slug = filename, preserved exactly from today's post filenames so URLs don't change.

## Migration (the load-bearing risk)

A one-time script (`scripts/migrate-to-astro.mjs`, deleted after use) converts each
`site/posts/<slug>.html` → `src/content/posts/<slug>.md`:
1. Extract `title` from `<h1>`, `pubDate` from `data-date`, `excerpt` from the existing
   index/feed entry (so listing text is identical).
2. Extract the body (the `<article>` inner HTML, minus `<h1>` and template chrome like
   `post-end`, `post-nav`, `post-subscribe`).
3. Convert body HTML → markdown with **turndown**, configured to preserve `<br>` as hard
   line breaks (critical — the prose relies on intra-paragraph line breaks) and to map
   `<p class="asterism">⁂</p>` → a thematic break.

### Verification gate (must pass before cutover)
For every post, build the Astro site and **diff the rendered article HTML against the
current live HTML's article body**, normalized for the known wrapper differences. Bodies
must be equivalent (same paragraphs, line breaks, emphasis, links). Any post whose rendered
body diverges is fixed (or its markdown hand-corrected) before proceeding. This is the same
gate that de-risked the Keystatic round-trip; with only 28 posts it is complete coverage,
not a sample. The WordPress export (`rascunhos.WordPress.2026-01-13.xml`) is available as a
cross-check source if a conversion is ambiguous.

## Design reproduction

The v2 CSS is reused unchanged. Astro components reproduce the exact HTML structure the v2
templates emit, so the same selectors apply:
- **Homepage rhythm:** posts sorted by `pubDate` desc → 1 featured (`post-featured` +
  `kicker` "o mais recente") + 2 recent (large) + the rest as the "mais rascunhos" archive
  list, 10 per page, paginated identically.
- **Post page:** `post-content`, `⁂` asterism breaks, prev/next `post-nav` (anterior/
  seguinte by date), per-post `post-subscribe`, floating subscribe button.
- **Author page** (`o-culpado/`), **footer**, **GoatCounter**, **Buttondown** forms — all
  reproduced from the current markup.

## RSS

`rss.xml.ts` via `@astrojs/rss` emits `feed.xml`-equivalent output at the same URL, same
item shape (title, link to `/posts/<slug>.html`, guid, pubDate, description=excerpt),
newest first, drafts excluded.

## Keystatic CMS

`keystatic.config.ts`: collection `posts`, `slugField: title`, `path:
src/content/posts/*`, `format: { contentField: 'content' }`, `fields.markdoc` body, fields
mirroring the Zod schema. Storage conditional: `local` in dev (no auth, for building/
testing) with a `PUBLIC_KEYSTATIC_STORAGE=github` override to run the setup wizard locally;
`github` (repo `hhpmacedo/rascunhos_blog`) in production. Admin at `/keystatic`; only
repo-write users (Hugo) can publish. **Authoring caveat:** plain markdown only — Markdoc
tags would render as literal text (Astro renders `.md` as CommonMark).

## Deployment & cutover

rascunhos.blog is **already served by Vercel** (`curl -I https://rascunhos.blog` →
`server: Vercel`; domain `rascunhos.blog` + `www` on the `rascunhos-blog` Vercel project; no
GitHub Pages CNAME). So cutover needs **no DNS change** — only a change to what the Vercel
project builds. Steps:
1. Build everything on `feat/astro-keystatic-cms`; Vercel auto-builds a **preview**.
2. Validate on the preview: every `/posts/<slug>.html` resolves, homepage/pagination/RSS/
   author parity, visual diff vs. production.
3. Replace deploy plumbing: update `vercel.json` (drop `outputDirectory: site`; let Astro's
   Vercel adapter define output) and **remove `.github/workflows/deploy.yml`** (the now-
   redundant GitHub Pages workflow — Pages does not serve the apex, so this only removes dead
   plumbing).
4. Merge to `main` → production (same Vercel project, same domain). Then the GitHub-App setup
   for Keystatic (interactive, Hugo): run the wizard **locally**
   (`PUBLIC_KEYSTATIC_STORAGE=github npm run dev`), set the 4 env vars in Vercel, redeploy,
   verify a live edit.

Zero-downtime: the live site is untouched until the Vercel build flips to Astro, and all
URLs + the feed are preserved.

## Risks & mitigations

| Risk | Mitigation |
| --- | --- |
| HTML→markdown body fidelity (esp. `<br>` line breaks, `⁂`) | turndown config + per-post rendered-HTML equivalence gate over all 28; WordPress XML cross-check |
| URL breakage | literal `.html` route preserves every path; preview-verify all 28 resolve |
| Two deploy pipelines fighting (Pages + Vercel) | remove the Pages workflow + confirm DNS before cutover |
| Keystatic body re-serialization changing output (known) | same gate as hugo-website; first edit per post restyles frontmatter but is render-neutral |
| `output: static` + Keystatic server routes | proven in hugo-website; fall back to `output: server` if needed (public pages still prerender) |

## Out of scope (YAGNI)

- No redesign — reproduce v2 exactly.
- No categories/tags/search unless the current site has them (it doesn't).
- No RSS-to-email automation (separate backlog item).
- No multi-author/roles — single author.
